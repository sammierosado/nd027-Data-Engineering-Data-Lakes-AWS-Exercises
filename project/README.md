# STEDI Human Balance Analytics Data Lakehouse Project

## Project Overview

This project implements a data lakehouse solution on Amazon Web Services (AWS) for STEDI, a company developing a Step Trainer device and a companion mobile application. As a Data Engineer, the primary goal was to extract, transform, and curate sensor data from the Step Trainer device and customer data from the mobile app into a structured format suitable for training a machine learning model. A core consideration throughout the pipeline was user privacy, ensuring only data from customers who explicitly consented to research was used.

## Problem Statement

STEDI collects data from three main sources:
1.  **Customer Records:** From fulfillment and website (contains PII, consent information).
2.  **Accelerometer Records:** From the mobile app (motion data, linked by user email).
3.  **Step Trainer Records:** From the device (motion sensor data, linked by serial number).

Key challenges addressed:
* **Data Privacy:** Only data from customers who agreed to share for research purposes (`shareWithResearchAsOfDate` is not null) should be used for ML model training.
* **Data Quality & Deduplication:** The initial customer data had duplicate entries (same email) and a known defect where fulfillment used non-unique serial numbers for customers.
* **Schema Consistency:** Ensuring critical columns like `timestamp` were correctly propagated and recognized throughout the pipeline.
* **Complex Joins:** Linking data across different sources (customer, accelerometer, step trainer) using various keys (email, serial number, timestamp) to create a unified dataset for machine learning.
* **Spark vs. Athena Behavior:** Understanding and accounting for differences in join results between AWS Glue (Spark) and Amazon Athena (Presto) when dealing with non-unique join keys.

## Solution Architecture

The solution leverages AWS services to build a multi-zone data lakehouse:

* **Amazon S3:** Used as the data lake storage for all raw, trusted, and curated data.
* **AWS Glue:** Utilized for serverless ETL (Extract, Transform, Load) jobs written in PySpark to move and transform data between zones. AWS Glue Crawlers are used to infer schemas and register tables in the Glue Data Catalog.
* **AWS Glue Data Catalog:** A central metadata repository storing table definitions, schemas, and locations, enabling services like Athena and Glue jobs to discover and query data.
* **Amazon Athena:** Used as a query engine for ad-hoc analysis and verification of data at each stage of the pipeline.
* **AWS IAM:** Manages permissions for Glue jobs and crawlers to access S3 and Glue Data Catalog resources.

**Data Flow (Zones):**
1.  **Landing Zone:** Raw, unprocessed data from source systems (JSON format).
2.  **Trusted Zone:** Cleaned, filtered, and deduplicated data from the landing zone, adhering to privacy and basic quality rules (Parquet format).
3.  **Curated Zone:** Highly refined, aggregated, and joined data from the trusted zone, optimized for specific analytical or machine learning use cases (Parquet format).
